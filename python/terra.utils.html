terra.utils.logger.connect_log_terra_version(app_name, terra_prefix=None)

Helper function to add the commit name of your app to the terra logs for your app.

Parameters:

  • app_name (str) – Your app name, used in the the log messages.

  • terra_prefix (str, optional) – The environment variable prefix. Defaults to "TERRA_"+app_name.upper()

Example

In your app __init__.py file, you should include the following:

from terra.utils.logger import connect_log_terra_version
connect_log_terra_version('AppName')

See also

terra.utils.logger.log_terra_version(sender, signal, app_name=None, terra_prefix=None, **kwargs)

Log the version of Terra App

Uses the logger configured dispatcher to log the Terra App version used to create the environment. Since this can be a complicated mixture of code versions, the Terra App version is logged by the main controller and by each service runner. This will capture a wide array of scenarios from development environment to fully containerized deploy.

Keyword Arguments:

  • app_name (str) – Name of app that appears in the logs

  • terra_prefix (str) – Name of environment variable prefix used

The following environment variables are needed to determine the git version:

Variables:

  • {terra_prefix}_IMAGE_COMMIT (str) – This should be set in the app’s image, and should be the output of git_commit_description.

  • {terra_prefix}_DEPLOY_COMMIT (str) – This should be set in the deploy image, and should be the output of git_commit_description.

  • {terra_prefix}_CWD (str) – When not running in the deploy image, the location of the app source code directory is used to determine the current version in the controller.

A typical log will include these lines to tell you what version is running:

2024-01-11 08:39:53,820 (computer.example.com:controller): INFO/MainProcess - __init__.py - Terra AppName version: heads/main-0-g5e17bc5
...
2024-01-11 08:40:03,382 (13ab62e95a44:runner): INFO/MainProcess - __init__.py - Terra AppName Runner version: heads/main-0-g5e17bc5

Examples

Details in the versions can be used to interpret a variety of situations, such as:

  • Development environment synced and running on main (as shown above):

    • Terra AppName version: heads/main-0-g5e17bc5

    • Terra AppName Runner version: heads/main-0-g5e17bc5

  • Development environment where just sync was run off of main, but then files were modified but not committed when running

    • Terra AppName version: heads/main-0-g5e17bc5

    • Terra AppName Runner version: heads/main-0-g5e17bc5-dirty

  • Development environment where just sync was run off sha 7b227ce, but then a different commit was used when running

    • Terra AppName version: heads/example_branch-0-g7b227ce

    • Terra AppName Runner version: heads/example_branch-0-g305bcbb

  • Fully containerized:

    • Terra AppName Deploy version: heads/main-0-g5e17bc5

    • Terra AppName Runner version: heads/main-0-g5e17bc5

    • When ever the deploy image is used, the word “Deploy” will appear.

    • In fully containerize it would be highly irregular for the sha’s to differ

    • In official deploy images, they should never be dirty

terra.utils.path module

terra.utils.path.logger = <Logger terra.utils.path (WARNING)>

Terra has multiple zones, the “controller”, the “compute”, the “executor”, and in the case of celery, the “task_controller” zone. Each zone has the potential to have the same files represented by different paths; e.g. volume binds in containers.

These utilities will help with handling those path translations.

terra.utils.path.patch_volume(value, volume_map, container_platform='linux')

Translate path value according to volume_map.

Parameters:

  • value (str) – Path value on the host

  • volume_map (list of tuple of str) – List of tuples. Each tuple contains two strings of the form (host_str, container_str).

  • container_platform (str, optional) – String specifying container platform, linux or windows. Defaults to linux.

Returns:

Translated path value in the container. If translation is not possible, function returns the original value.

Return type:

str

terra.utils.path.pathlib_map(volume_map, container_platform='linux')

Translate volume mapping from strings to pathlib objects. Source paths on the host system are instantiated as Concrete paths allowing use of the pathlib.Path.resolve() function. Paths in the container are instantiated as Pure paths of the appropriate type.

Parameters:

  • volume_map (list of tuple of str) – List of tuples. Each tuple contains two strings of the form (host_str, container_str).

  • container_platform (str, optional) – String specifying container platform, linux or windows. Defaults to linux

Returns:

List of tuples. Each tuple contains two pathlib objects of the form (host_pathlib, container_pathlib). host_pathlib objects are Concrete paths suitable for the current host OS. container_pathlib objects are Pure paths suitable for the specified container_platform.

Return type:

list of tuple of pathlib objects
